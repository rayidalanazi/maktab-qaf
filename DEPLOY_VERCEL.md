# نشر قاف على Vercel — لتجربة تعدّد المكاتب والسب دومين

GitHub Pages **ساكن** (لا سيرفر) فلا يدعم `proxy.ts` ولا المسارات الديناميكية ولا السب دومين.
Vercel يشغّل Next.js بسيرفر حقيقي، فيعمل كل ذلك. هذا الدليل خطوة بخطوة.

الكود **جاهز** للسيرفر: بناء وضع السيرفر ينجح، `proxy.ts` يلتقط السب دومين، و`/t/<أي-مكتب>/`
يُرسم ديناميكياً (لا حاجة لبناء صفحة لكل مكتب).

---

## المرحلة 1 — انشر على Vercel (≈ 3 دقائق، مجاني)

1. ادخل https://vercel.com → **Continue with GitHub**.
2. **Add New… → Project** → استورد المستودع `rayidalanazi/maktab-qaf`.
3. Framework Preset: **Next.js** (يُكتشف تلقائياً). اترك الإعدادات الافتراضية.
   - **مهم:** لا تُضِف متغيّر البيئة `STATIC_EXPORT` (نريد وضع السيرفر، وهو الافتراضي).
   - لا يلزم أي متغيّر بيئة آخر — مفاتيح Supabase وGoogle مضمّنة كافتراضيات في الكود.
4. **Deploy**. ستحصل على رابط مثل `maktab-qaf-xxxx.vercel.app`.

## المرحلة 2 — فعّل دخول Google على الرابط الجديد (دقيقة)

دخول Google يتطلب أن يكون «أصل» الموقع مُصرّحاً به في Google Cloud:

1. https://console.cloud.google.com → المشروع `rosy-etching-496616-j0`.
2. **APIs & Services → Credentials** → افتح عميل OAuth `106593971410-…`.
3. تحت **Authorized JavaScript origins** أضف الرابط بالضبط:
   ```
   https://maktab-qaf-xxxx.vercel.app      (استبدله برابطك الفعلي من Vercel)
   ```
4. **Save**. (الدخول بالإيميل/كلمة المرور يعمل بدون هذه الخطوة.)

> Supabase: لا يحتاج تغييراً — نستخدم تدفّق Google id-token (بدون redirect) + إيميل/كلمة مرور.

## ✅ اختبار الفكرة فوراً (بدون دومين)

على Vercel المسارات ديناميكية، فجرّب — كل واحد يعرض مكتباً مختلفاً على **سيرفر حقيقي**:
```
https://<رابطك>.vercel.app/t/raed/
https://<رابطك>.vercel.app/t/meeem/      ← يعمل رغم أنه لم يُبنَ مسبقاً
https://<رابطك>.vercel.app/admin/
```
هذا يُثبت تعدّد المكاتب + المسار الديناميكي على سيرفر. (سجّل الدخول لرؤية بيانات مكتبك الحقيقي.)

---

## المرحلة 3 — السب دومين الحقيقي `meeem.qaf.sa` (يتطلب دومين)

`*.vercel.app` لا يعطي wildcard، فالسب دومين الحقيقي يحتاج دومينك الخاص:

1. **سجّل دومين** (qaf.sa من nic.sa، أو `.com` أسرع).
2. **Vercel → Project → Settings → Domains** أضف:
   - `qaf.sa` (الأصل) → صفحة التسويق
   - `*.qaf.sa` (wildcard) → كل المكاتب  ⚠️ النطاق الـwildcard على Vercel يتطلب **خطة Pro**.
     (على المجانية: أضف نطاقات مفردة يدوياً مثل `meeem.qaf.sa` للاختبار.)
3. **DNS** عند مزوّد الدومين: سجل `*` (CNAME/ALIAS) → نقطة Vercel.
4. **Vercel → Settings → Environment Variables** أضف:
   ```
   NEXT_PUBLIC_BASE_DOMAIN = qaf.sa
   ```
   (يجعل `proxy.ts` يلتقط السب دومين من دومينك بدل qaf.sa الافتراضي.)
5. أعد النشر. الآن `meeem.qaf.sa` → يوجّه داخلياً لـ `/t/meeem` ويبقى الرابط `meeem.qaf.sa` في المتصفح.

### ⚠️ نقطة مهمة: Google + السب دومين

Google **لا يسمح بـ wildcard في الأصول المصرّح بها** — فلا يمكن إضافة `*.qaf.sa`. الحلّ المعتمد:
- تمركز تسجيل الدخول على أصل واحد (مثل `qaf.sa` أو `app.qaf.sa`)، ثم مشاركة جلسة Supabase
  عبر **كوكي بنطاق `.qaf.sa`** (بدل localStorage الخاص بكل أصل).
- هذا تحويل بسيط لطبقة المصادقة (`@supabase/ssr` بكوكي على `.qaf.sa`) — أنفّذه لك وقت ننتقل
  للسب دومين الإنتاجي. للاختبار الحالي (مسار `/t/slug/`) لا يلزم لأنه أصل واحد.
